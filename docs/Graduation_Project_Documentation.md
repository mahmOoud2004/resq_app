# ResQ Graduation Project Documentation

## 1. Project Title

**ResQ: Smart Emergency Response and AI Healthcare Assistant**

## 2. General Introduction

ResQ is a mobile application developed using Flutter.
The main purpose of this project is to support people during emergency situations.
The application gives users a fast and simple way to ask for emergency help.
At the same time, it supports drivers, administrators, and users who need health guidance.

This project was built as a graduation project with a real social goal.
It is not only a technical exercise.
It is a practical system that tries to solve an important problem in daily life.
Many people may face accidents, medical emergencies, or sudden dangerous situations.
In such moments, every second is important.
Because of that, the project focuses on speed, clarity, and reliability.

ResQ combines more than one idea in one platform.
It includes emergency request management, user and driver communication, live location tracking, smart health notifications, and AI healthcare support.
This combination gives the project a stronger value.
The application is useful in urgent moments, but it is also useful in normal daily health care.

The project tries to answer an important question:
How can a mobile application make emergency support faster, easier, and smarter?

The answer in this project is based on modern mobile development tools, clean application design, and role-based workflows.
The system is designed to be simple for users, helpful for drivers, and manageable for administrators.

## 3. Background of the Problem

Emergency support is one of the most important services in any society.
When a person is in danger, the quality and speed of the response can make a big difference.
In many cases, people still depend only on phone calls.
Phone calls are useful, but they have many limitations.

For example, the user may not be calm enough to explain the problem.
The user may not know the exact location.
The person receiving the call may not understand the emergency clearly.
In some cases, the line may be busy or the communication may be weak.
All these issues can cause delay.

Another problem is the lack of direct live tracking.
If the user sends a request, it is helpful to know:

- Has the request been accepted?
- Is the driver moving?
- How far is the driver?
- What is the current status of the case?

Without clear tracking, the user may feel confused or stressed.
The driver may also need a clear map and navigation support.

There is also another related issue.
Many users need regular health support even when there is no emergency.
Some people live with chronic diseases such as diabetes, hypertension, asthma, or heart disease.
These users may need reminders, health tips, and a simple way to store personal health information.

In addition, modern users now expect digital services to be smart.
They want applications that are not only basic tools, but also intelligent assistants.
Because of this, adding AI support makes the project more useful and more modern.

This is the context that led to the development of ResQ.
The project tries to reduce communication problems, improve response speed, provide live tracking, and add health support inside one mobile application.

## 4. Problem Statement

## 13. General Architecture of the Project

The project follows a feature-based structure.
This means that code is organized by feature instead of putting all logic in one place.

This structure makes the project easier to understand.
It also helps reduce confusion when the app becomes larger.

At a high level, the project contains:

- `main.dart`
- `config`
- `core`
- `features`

### 13.1 Main Entry Point

The application starts from `main.dart`.
This file prepares the Flutter app, loads environment values, applies theme setup, and starts the router.

### 13.2 Config Layer

The `config` folder handles routing and screen paths.
It helps the app know where each screen belongs and how to move between screens.

### 13.3 Core Layer

The `core` folder contains common reusable elements.
It includes:

- constants
- theme
- storage
- network
- error handling

This layer is shared between many features.

### 13.4 Features Layer

The `features` folder contains the actual business modules.
Each module has its own screens, logic, models, and data files.

This is one of the strongest parts of the project structure.
It keeps the code modular and easier to maintain.

## 15. Authentication Module

The authentication module controls user entry into the system.
It includes login, signup, OTP verification, forgot password, and reset password.

This module is important because the system has different user roles.
The app must know whether the person is a normal user, a driver, or an administrator.

### 15.1 Main Authentication Screens

- Login Screen
- Signup Screen
- OTP Screen
- Forget Password Screen
- Reset Password Screen

### 15.2 Main Authentication Logic

The authentication flow works through API calls.
When the user logs in, the app sends user credentials.
If the response is successful, the token is saved locally.
Then the app checks the role and opens the correct route.

OTP verification is used in some flows.
This adds another level of identity confirmation.

### 15.3 Value of This Module

This module gives the app control and security.
It also makes the user journey more organized.

## 16. Role-Based Navigation

One of the important design ideas in the app is role-based navigation.
This means the app does not treat every user the same way.

After authentication, the app checks the role.
Then it redirects the person to the proper area.

For example:

- user goes to the main home flow
- driver goes to the driver interface
- admin goes to the admin panel

This design improves usability.
It also reduces mistakes because every person sees only what matches their tasks.

## 33. Application Workflow for a Normal User

The normal user flow can be explained in detail:

1. The user opens the app.
2. The splash and startup logic begin.
3. The user logs in or creates a new account.
4. If OTP verification is needed, the user completes it.
5. The app checks the role.
6. The user enters the home flow.
7. The app reads current location.
8. The user presses the emergency button.
9. The user chooses the service type.
10. The request is sent to the backend.
11. The system checks request state.
12. The user tracks the driver on the map.
13. The trip is completed.

This full flow is one of the strongest examples of how the application solves a real problem.

## 34. Application Workflow for a Driver

The driver flow can also be explained step by step:

1. The driver logs in using the driver account.
2. The app opens the driver area.
3. The driver sets status to online.
4. The app loads available nearby requests.
5. The driver reviews a request.
6. The driver accepts or rejects it.
7. If accepted, the driver opens the map flow.
8. The driver moves toward the user.
9. The location can be updated.
10. The driver completes the request.

This flow gives the driver a clear operational experience.

## 35. Application Workflow for an Admin

The admin flow is more focused on monitoring and management:

Because the project contains many features, organizing the code was a real challenge.

### 37.2 Multi-Role Challenge

The system has users, drivers, and admins.
Each role has different needs and different screens.

### 37.3 Live Tracking Challenge

### 37.5 User Experience Challenge

The app had to stay simple.
In emergency systems, too many steps or unclear screens can reduce usability.
So the interface needed to stay direct and easy.

## 38. Solutions Used to Handle the Challenges

Several solutions were applied during the project.

### 38.1 Feature-Based Structure

Dividing the code by feature improved readability and maintainability.

### 38.2 Bloc and Cubit

### 38.4 Better Route Safety

### 38.5 Cleaner Driver State
